Issue Tracker: A full-stack app

built with MERN

Overview

Hey there, future Tim here. This is a post-mortem of my Issue Tracker app. I'm writing this after having a few more projects under my belt. Here, I'll be touching on more of the nuts and bolts of the project. For something more sophisticated check out my write-up on my e-commerce project. I've learned a lot since I worked on this code, but this project was a great exercise.

Here is what I wanted to accomplish with this project:

  • Implement React and Redux for the front-end.
  • Use Node, Express, and Mongoose on the back-end.
  • Deploy a finished project.
  • Familiarize myself with a code editor and GitHub.

This was my very first project that I create entirely on my own after completing the FreeCodeCamp.org cirriculum. I was excited and nervous -- staring at a blank canvas was quite daunting. Time to put it all together: React, Node, Express, and Mongoose.

I decided to create an issue tracker that took inspiration from a cap-stone project which I completed towards the end of my time with Free Code Camp. The project would work off of a foundational concept while adding in more functionality. I took the idea of a basic CRUD app and added user account creation, project grouping, and shareability between users.

Challenges

Let's start things off by setting up a back-end route. To note, I didn't separate routes and controllers on this project -- something I would do now that I have more experience. 

const router = require('express').Router()

//Load input validation
const validateSubmitInput = require('../validation/submitIssue')

//Load project model
const Issue = require('../models/Issue')
  
router.post('/:user/:project/:issue', (req, res) => {
  //Validate inputs
  const { errors, isValid } = validateSubmitInput(req.body)
  //return errors if invalid
  if (!isValid) {
    res.status(400).json(errors)
  } else {
    //check DB by title
    Issue.findOne({ user: req.params.user, project: req.params.project, title: req.body.issue_title }).then(issue => {
      if (issue) {
        res.status(400).json({ title: 'That issue already exists' })
      } else {
        const newIssue = new Issue({
          user: req.params.user,
          project: req.params.project,
          issue_title: req.body.issue_title,
          issue_text: req.body.issue_text,
          created_on: new Date(),
          created_by: req.body.created_by,
          assigned_to: req.body.assigned_to || '',
          status_text: req.body.status_text || ''
        })

        //save the project in the DB
        newIssue
          .save()
          .then(issue => res.json(issue))
          .catch(err => console.log(err))
      }
    })
  }
})

I've left off my model and the validator to save space. You can see them here and here. One thing I want to call out is that my back-end code for this project is doing some nested actions. Knowing what I know now, if I were to implement testing I would write things a little bit differently in order facilitate that process.

Let's take a look at a front-end action that hits this end point: 

//Create issue
export const createIssue = newIssue => dispatch => {
  axios
    .post(`/api/issues/${newIssue.user}/${newIssue.project}/${newIssue.issue_title}`, newIssue)
    .then(res => {
      dispatch({
        type: CREATE_ISSUE,
        payload: res.data
      })
      dispatch(clearErrors())
    })
    .catch(err =>
      dispatch(returnErrors(err.response.data, err.response.status))
    )
}

We fire off a higher order function which makes a post request to our back-end, it then calls the appropriate reducer(s).

Over in our reducer we're adding the newly created issue into our global state so that we can display it via its React component. We do this by combining the existing state (grabbed with the spread operator) and the new issue in an array: 

const initialState = {

I've trimmed out some of the state and form parts for length's sake. I hope these code blocks help illustrate an end to end trip through the app.

One of the challenges I encountered during this project was needing to simultaneously enable an issue to be edited while disabling all non-selected issues. To accomplish this I created a global edit action/reducer to put the issue into edit mode, in the mean time I compared the issue's ID with the global state's issues. These paired with conditional rendering did the trick.

On to deployment. A technique I implemented to have an smoother deployment was to nest the client folder within the back-end project. Each "project" then has its own package.json. This gives us two advantages. First, Inside of the client's package.json we assign a key "proxy" to the same port that we're using for our back-end. Now we can run the React front-end and the server via a script with Concurrently like so: 

"dev": "concurrently \"npm run server\" \"npm run client\""

Second, we can create a postbuild script which installs all of our client dependencies and creates our React production build. After all that we can push to Heroku and cross our fingers.

Once we succeed with our Heroku push all that's left is to set our environment variables via our Heroku dashboard!


Takeaways

React & Redux: I'm really starting to enjoy and get comfortable with React. Even though this project was written with class components and I've since moved on to hooks, I was able to get in valuable general React practice through repetition. Also, now that I've experimented on a few more projects, I like the workflow of Redux despite the boiler-plate needed to get things going.

Back-end: I now have two similar styles of user authentication under my belt. The first was using bcrypt to handle encryption and the second was using crypto which is now built into Express. I've also both built the encryption method into the user model as well as having it be part of the auth controller. The encryption as part of Mongoose model does feel a bit cleaner, but both styles seemed to work well.

Deployment: All things considered, I had a really smooth experience with Heroku. It was a much more stream-lined process than other platforms that I looked into.

Overall, even though I would do some things differently, I learned a lot while making this app. I'm happy with the overarching structure of the project -- admittedly my front-end variables could use some destructuring and tidying up. Thanks for reading! Feel free to check out the full repository.
